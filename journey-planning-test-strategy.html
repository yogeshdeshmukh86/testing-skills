1. Product Overview

The Journey Planning App enables urban commuters and travellers to plan multimodal trips across public transport networks. It uniquely addresses the first mile (home to transit stop) and last mile (transit stop to destination) problem by integrating walking, cycling, micro-mobility, and ride-hailing options into a unified itinerary.

 

A core differentiator is real-time capacity and occupancy planning: the app surfaces live vehicle crowding data, recommends less-crowded alternatives, and allows users to make informed decisions about departure times based on predicted occupancy levels.

 

Key User Journeys

•       Plan a multimodal trip with first/last mile alternatives

•       View real-time vehicle occupancy and select preferred departure

•       Receive live disruption alerts with automatic re-routing

•       Book or unlock a micro-mobility asset (bike/scooter) for last mile

•       Navigate turn-by-turn through a planned itinerary

 

Product Tier Classification

T1 – Consumer App. Broad public audience, app store distribution, iOS and Android, diverse device fleet, real-time data dependencies, location-critical features, privacy-sensitive (location history, travel patterns). Full 6-dimension test coverage applies.

•       Accessibility compliance (WCAG 2.1 AA, platform a11y APIs)

•       Ticketing and payment flows (covered by separate fintech strategy)

•       Verify occupancy data renders correctly under real-time update load

•       Confirm graceful degradation when real-time feeds are unavailable

•       Ensure app meets WCAG 2.1 AA and platform accessibility standards

•       Validate security of location data storage and transmission

•       Verify battery and performance within acceptable thresholds on low-end devices

 

 

3. Test Dimensions

3.1  GPS, Location & Sensors

This is the most critical dimension for a journey planning app. The entire first/last mile value proposition depends on accurate, timely location data. Testing must cover both simulation (for regression) and real-world field testing (for truth).

 

First Mile & Last Mile Specifics

•       Walking route accuracy: path snapping, off-route detection threshold

•       Mode detection: does the app correctly detect when user switches from walking to cycling or transit?

•       Indoor-to-outdoor GPS handoff at station entrances — loss of signal and re-acquisition

•       Time to first fix (TTFF) cold/warm: must acquire within 10 seconds for good UX

•       Location spoofing resilience: users faking location to claim micro-mobility assets

•       Background location: continue tracking during active navigation; correct behaviour when backgrounded

•       Geofencing accuracy: arrival detection for stops, micro-mobility docking zones

 

Capacity & Occupancy

•       Real-time feed latency: occupancy data freshness (target: <30s stale)

•       Feed interruption: occupancy widget gracefully shows 'data unavailable' without crash

•       Data accuracy validation: cross-check displayed occupancy vs operator ground truth

 

Tools

•       Xcode Simulator: GPX route playback for regression on iOS

•       Android Emulator: GPS mock injection via ADB

•       LocationSimulator (macOS): live route simulation for iOS

•       SensorLog / Physics Toolbox: field data capture for comparison

•       QGIS / Google Maps: ground-truth route validation

 

3.2  Network & Connectivity

Journey planning is inherently real-time. Network quality is highly variable in transit environments — underground stations, tunnels, and crowded areas all cause signal degradation. The app must handle these gracefully without losing the user's journey context.

 

Transit-Specific Network Scenarios

•       Underground / tunnel: complete signal loss mid-journey — navigation continues offline

•       Station concourse: high-density WiFi and cellular contention

•       Cross-border travel: roaming handoff, potential data restrictions

•       Real-time feed polling: occupancy and disruption updates on 3G vs 4G vs offline

•       Push notification delivery: disruption alerts when app is backgrounded on poor connectivity

 

Offline Behaviour

•       Last cached journey plan must be accessible with no network

•       Offline navigation: pre-downloaded map tiles continue turn-by-turn routing

•       Stale data indicators: clearly communicate when occupancy data is from cache

3.4  Security & Privacy

Journey planning apps accumulate highly sensitive behavioural data — home address, work address, daily movement patterns, preferred departure times. GDPR applies fully. Location history is a special category of sensitive data in many legal frameworks.

 

Location Data Handling

•       Location history must not be stored on device beyond session unless user explicitly opts in

•       Cached journey plans: scrub precise coordinates from cache after configurable retention period

•       No location data in crash reports or analytics logs

•       'Precise vs approximate' location: app must function with approximate location for journey planning; precise required only for live navigation

 

Occupancy Feed Security

•       API authentication: operator occupancy feeds protected by token, not open endpoints

•       Certificate pinning for real-time data feeds to prevent MITM on crowding data

•       No PII in occupancy API responses — aggregate counts only

 

Micro-Mobility Integration

•       BLE unlock tokens: short-lived, device-bound — cannot be replayed

•       QR codes: server-side validation, single-use

•       Deep links: bike booking confirmation links cannot be triggered without authentication

 

Compliance

•       GDPR Article 25: privacy by design — data minimisation by default

•       GDPR Article 17: data deletion — 'delete my journey history' flow tested end-to-end

•       Consent: location 'always on' requires explicit informed consent with clear opt-out

 

3.5  Accessibility (a11y)

Public transport apps serve a broad public including users with visual, motor, and cognitive disabilities. This is both a legal requirement (EN 301 549 in the EU) and a critical equity consideration — many users depend on public transport precisely because they cannot drive.

 

Journey Planning UI

•       Screen reader: VoiceOver / TalkBack can navigate full journey planning flow without visual reference

•       Occupancy indicators: crowding level must not be conveyed by colour alone — add icon + text label

•       Map: map tiles are decorative (marked as accessibility-hidden); all key info in text layer

•       Dynamic Type / Font Scale: itinerary cards and stop lists remain readable at 200% font scale

 

Navigation Mode

•       Turn-by-turn instructions announced via accessibility live regions (not just visual arrows)

•       Haptic feedback on approaching turn / mode change

•       Motor: all navigation controls reachable with one thumb (bottom-of-screen action zone)

•       Reduced Motion: map animations and route drawing transitions suppressed when enabled

 

Real-Time Updates

•       Disruption alerts: announced immediately via VoiceOver/TalkBack live region on arrival

•       Occupancy updates: only announced if user has occupancy panel focused — not on every refresh

7. Entry & Exit Criteria

Entry Criteria (before testing begins)

•       Build is stable: no known crash-on-launch defects

•       Test environments configured: mock occupancy feed, GPS simulation ready

•       Device matrix available: minimum Tier 1 devices provisioned

•       Test data ready: representative journey plans, occupancy scenarios, disruption events

 

Exit Criteria (before release)

•       All Tier 1 device / OS combinations pass core journey planning E2E flow

•       Zero P1/P2 defects open in GPS, occupancy, or security dimensions

•       Performance targets met on Tier 1 devices (cold start, battery drain, frame rate)

•       WCAG 2.1 AA audit passed; no screen reader blockers on core flows

•       GDPR deletion flow validated end-to-end

•       Security scan: no critical or high findings unresolved

•       Field testing completed on at least 2 real transit routes
